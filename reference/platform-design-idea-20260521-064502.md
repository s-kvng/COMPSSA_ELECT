## Problem Statement

The COMPSSA department runs internal elections to select student representatives
for SRC portfolios and internal executive positions. The current paper/manual
process has produced disputed results — students and candidates have contested
vote counts, ECs have had their integrity challenged, and the department has no
mechanism to prove the final count was accurate. The department formally
requested a digital solution.

The platform must make election disputes impossible, not just unlikely. Three
guarantees must be provable to any skeptic: (1) only registered department
students voted, (2) no student voted more than once per category, (3) the
announced result matches the actual tally with no room for manual manipulation.

## Demand Evidence

- The department explicitly commissioned this platform — not speculative demand
- Prior elections: results were contested, recount demanded, EC reputation
  damaged, candidates felt cheated
- ECs took personal reputational heat when results were challenged
- The gap from "manual paper ballots" to "auditable digital platform" is the
  exact pain point the department wants eliminated

## Status Quo

Manual paper-based election process:
1. ECs prepare physical ballot sheets with candidate names per category
2. Students queue at a physical location, mark ballots, deposit in box
3. ECs manually count ballots after voting closes
4. Results announced verbally or via notice board
5. No audit trail, no way to recount independently, no proof of vote uniqueness

Failure modes experienced: disputed counts, inability to prove who voted,
accusations of ballot stuffing, manual errors in tallying.

---

## Target User & Narrowest Wedge

**Most desperate user: The EC (Electoral Commissioner)**
They are responsible for running a fair election and defending the result.
When results are disputed, they have no evidence to present. Their reputation
is on the line and they have no tools to protect it.

**Narrowest wedge:** A single election (one day, one set of categories) that:
- Only lets registered students vote
- Enforces one-vote-per-category
- Gives ECs a live dashboard they can screenshot as proof
- Announces results to all students after polls close

If this single election runs without dispute, the platform has achieved its goal.

## Premises (All Confirmed)

1. **Pre-populated accounts only** — ECs create all student accounts;
   no self-registration. Default password (e.g., student ID) + mandatory
   first-login password change.

2. **One vote per category, irrevocable** — Once a student votes in a
   category, that vote cannot be changed or withdrawn.

3. **EC-locked voting windows** — Start time and end time are set during
   election setup. Once voting is active, these cannot be changed.

4. **Staged transparency:**
   - During voting: ECs + HOD see full live counts; candidates see their
     own count only; students see voting progress (how many have voted)
     but NOT candidate tallies.
   - After voting closes: all users see full results.

5. **Privacy-preserving vote storage** — `voted_log` (who voted in which
   category) is NEVER joined with `candidate_tallies` (vote counts per
   candidate). No one — including admins or the developer — can look up
   which candidate a specific student voted for.

## Data Model

```
users
  _id, name, studentId, email, passwordHash, role (student/EC/HOD)
  isFirstLogin: bool
  departmentId: string

elections
  _id, name, description
  status: 'draft' | 'ready' | 'active' | 'closed' | 'published'
  startTime: timestamp, endTime: timestamp   ← immutable once status >= 'active'
  earlyClosedAt?: timestamp                  ← set if EC triggered early close
  createdBy: userId (EC)

categories
  _id, electionId, name (e.g. "SRC President")
  description

candidates
  _id, categoryId, studentId (ref → users)
  name, bio
  // photoUrl: V2 only — omitted from V1 schema entirely

voted_log                    ← PRIVACY PARTITION A: who voted in which category
  _id, voterId (ref → users)
  categoryId, electionId
  timestamp
  // Index: by_voter_category on (voterId, categoryId) — hot path for duplicate check

candidate_tallies            ← PRIVACY PARTITION B: vote counts without voter link
  _id, candidateId (ref → candidates)
  categoryId, count: number
```

**Privacy partition:** No application query joins `voted_log` and
`candidate_tallies`. The atomic mutation writes to both in one Convex transaction,
which means both rows share the same `_creationTime`. In low-voter-count
categories, a privileged admin with raw DB access could correlate by timestamp.
The guarantee is scoped to the application layer: "no application code can
deanonymize a ballot." This is the right guarantee for a department-scale system
run by a trusted EC; it is not cryptographic ballot secrecy.

**`departmentId` on users:** Omitted from V1 schema (YAGNI). Add it when
multi-department support becomes a real requirement.

## Vote Mutation (Core Logic)

```typescript
// convex/votes.ts — castVote mutation (runs on Convex server)
// This is the "indomitable" guarantee: atomic, server-enforced, no race condition
mutation castVote(args: { categoryId, candidateId }) {
  // NEVER accept voterId as an argument — always derive from session
  const identity = await ctx.auth.getUserIdentity()
  if (!identity) throw new Error("unauthenticated")
  const voter = await ctx.db
    .query("users")
    .withIndex("by_token", q => q.eq("tokenIdentifier", identity.tokenIdentifier))
    .unique()
  if (!voter || voter.role !== 'student') throw new Error("unauthorized")

  1. Verify election is active (status === 'active', now between start/end or before earlyClosedAt)
  2. Verify candidateId belongs to categoryId (guard against cross-category vote injection)
  3. Query voted_log by index by_voter_category(voter._id, categoryId):
     → If row exists: return { status: 'already_voted' }   ← not an error, safe to retry
  4. Within this Convex mutation (atomic, serializable by definition):
     a. Insert voted_log: { voterId: voter._id, categoryId, electionId, now() }
     b. Read candidate_tallies row for candidateId; patch with { count: existing.count + 1 }
        (Read-then-patch is safe here — Convex mutations are serialized; no concurrent interleave)
  5. Return: { status: 'ok' }
}
```

**On retry safety:** The first call returns `{ status: 'ok' }`. A second call
returns `{ status: 'already_voted' }`. Safe to retry on network failure.

**`candidate_tallies` initialization:** When an EC adds a candidate to a category,
the system inserts `{ candidateId, categoryId, count: 0 }` into `candidate_tallies`.
This happens in the `addCandidate` mutation, not at vote time. The `castVote`
mutation can assume the row exists (throws if not, which surfaces a setup bug).

**Cron job for election state transitions:**
Convex has no row-level triggers. State transitions driven by time require a
scheduled cron:
```typescript
// convex/crons.ts
crons.interval("advance election states", { minutes: 1 }, internal.elections.advanceStates)
// advanceStates: query elections where startTime <= now() && status === 'ready' → set 'active'
//                query elections where endTime   <= now() && status === 'active' → set 'closed'
```
ECs can also manually open/close from the dashboard, which bypasses the cron.

**Role-gated `electionResults` query:**
Only one election can be ACTIVE at a time (enforced by the `activateElection`
mutation which checks for existing active elections). The `electionResults` query
derives the caller's role from session and gates the response:
- role === 'EC' or 'HOD': return full live counts regardless of election status
- role === 'student': return counts only if status === 'published';
  return `{ status: 'voting_in_progress' }` while active.
  Additionally: if the caller has a row in the `candidates` table linked to their
  userId, also return their own `candidate_tally` count (always, even during voting).
  Check the `candidates` table — NOT `users.role` which has no 'candidate' value.
  (Candidates have `role = 'student'` in `users`; candidacy is a separate relationship.)

**`castVote` — additional guards:**
- Check `now() < args.endTime` server-side as defense-in-depth (the cron can lag
  up to 59 seconds; this closes that window).
- Check `voter.isFirstLogin === false` — a student who has not changed their
  default password should not be able to vote. Return `{ status: 'must_change_password' }`.
  This is server-enforced (not UI-only).
- All `addCandidate`, `removeCandidate`, `addCategory`, `removeCategory` mutations
  must assert `election.status === 'draft'` before writing.

## Auth Flow Detail

```
HOD account creation:
  → Created manually by the developer at deploy time (or by an EC using a
    privileged mutation scoped to role: 'HOD')
  → Same default password pattern as students: HOD's studentId, lowercased
  → isFirstLogin: true → same first-login password change flow
  → V1: one HOD account per deployment; no HOD self-registration

Session lifetime:
  → Convex Auth default session (no custom expiry in V1)
  → If session expires mid-vote, client retries castVote — safe because
    castVote is already safe to retry (returns 'already_voted' if duplicate)

EC creates student account (bulk CSV or manual)
  → CSV columns: name, studentId, email  (3 columns, header row required)
  → System sets: { role: 'student', isFirstLogin: true }
  → Default password: studentId value, lowercased, trimmed (e.g. "cs2023001")
     EC communicates this to students verbally or via class WhatsApp group
     (Out-of-band delivery is acceptable for department scale; no email send required in V1)
  → Bulk mutation: all-or-nothing per batch; EC sees per-row error report before commit

Student logs in for first time
  → POST /auth/login → success
  → Server detects isFirstLogin === true
  → Redirect to /first-login
  → Student sets new password (min 8 chars)
  → Server sets isFirstLogin = false
  → Redirect to /vote or /dashboard

Subsequent logins
  → Normal flow: email + password → session token → role-based redirect
```

## Election Lifecycle

```
DRAFT ──[EC configures categories + candidates + start/end time]──▶ READY
READY ──[startTime reached OR EC manually opens]──────────────────▶ ACTIVE
ACTIVE ──[endTime reached OR EC manually closes]──────────────────▶ CLOSED
CLOSED ──[results published to all students]──────────────────────▶ PUBLISHED

State machine rules:
  - DRAFT → READY: requires at least 1 category, 1 candidate per category
  - READY → ACTIVE: EC cannot change categories/candidates after this point
  - ACTIVE → CLOSED: `endTime` value is IMMUTABLE once active — cannot be edited.
    EC CAN trigger an early close (sets status to CLOSED immediately, stores
    earlyClosedAt timestamp). This does not modify endTime; it short-circuits
    the schedule. Requires explicit confirmation dialog ("Close election now?
    This cannot be undone.")
  - CLOSED → PUBLISHED: EC reviews results, then publishes
```

## Approaches Considered

### Approach A: All-Convex (Convex Auth) — CHOSEN
Single service: Convex handles auth, DB, and real-time subscriptions.
Bulk student import via custom admin mutation (CSV parsed in browser, sent
to Convex mutation as a batch). Convex transactions enforce vote atomicity.
Effort: M (human ~4 days / CC ~3 hours).

### Approach B: Clerk + Convex
Clerk for auth (admin SDK for bulk user creation), Convex for data and
real-time. Better auth UX but adds a second service and billing dependency.
Deferred — worth revisiting if Convex Auth bulk import is painful.

### Approach C: Custom password auth from scratch
Total control, zero external services. Rejected: you own the security surface,
too much risk for marginal gain at department scale.

---

## Recommended Approach

**Approach A: All-Convex.** The repo is already scaffolded with Convex. The
Convex transaction model is the exact primitive needed for the "indomitable"
vote guarantee — it's physically impossible to insert into `voted_log` without
atomically incrementing `candidate_tallies`. Real-time subscriptions are
built-in. Auth is sufficient for this use case.

The bulk import limitation (no CSV UI out of the box) is real but manageable:
build a simple CSV → Convex mutation admin tool in the first sprint.

**Implementation note — Convex Auth bulk user creation:** Convex Auth's
password provider owns password hashing; you cannot insert raw password hashes
directly into the users table. Bulk creation requires calling Convex Auth's
internal `createAccount` function inside a server action. **Caution:** this is
an internal API that is not in Convex's public docs and has changed between
`@convex-dev/auth` releases. Before starting: verify the exact internal module
path for your installed version. Approach B (Clerk) is the fallback if this API
proves unstable. Convex mutations have document limits — batch in groups of max
50 students per call, client loops over batches. Also: after the first-login
password change, refresh the session token (re-authenticate) rather than just
redirecting — stale tokens from the default-password session should not persist.
Expect ~1.5 extra days vs. a plain-DB approach. If this proves painful, revisit
Clerk (Approach B).
