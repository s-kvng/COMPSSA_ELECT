## Vision

### 10x Check
The department currently runs one election per year with paper ballots and disputes the result every time. The 10x version of this platform is not just "no disputes" ‚Äî it's "the department treats this election infrastructure the way AIMS treats its server: it runs, nobody thinks about it, everybody trusts it." That means: students log in once, see their ballot, vote in two minutes, close the tab. ECs open the dashboard, watch numbers climb in real-time, screenshot it when voting closes, publish, done. No recount requests. No "did you tamper with it" accusations. Just a result that everyone accepts because the math is visible and the process was locked.

### Delight Opportunities
1. Student sees a check mark per category as they vote ‚Äî no reload, no confirmation page, just instant feedback that their vote registered.
2. EC exports the voter audit log in one click and sends it as a file, not a screenshot, to any skeptic.
3. Public results link that works without login ‚Äî EC sends one WhatsApp message with a link and the whole department can see results without needing their login credentials.
4. Candidate dashboard that shows their tally climbing in real time ‚Äî no F5 required, Convex subscription updates live.
5. First-login flow that feels intentional: "Set your password before you can vote." Clear reason, not a friction bump.

## Accepted Scope (added to this plan)

### Core design doc scope (reproduced for reference)
All four roles are in scope: **Student**, **Candidate** (student with candidateProfile), **EC** (Electoral Commissioner), **HOD** (Head of Department, read-only).

**Screen map:**
```
AUTH
  /login                     All roles
  /first-login               Force password change (isFirstLogin === true)

STUDENT
  /vote                      Active election: category list + voting progress
  /vote/[categoryId]         Candidate selection + confirm vote
CANDIDATE
  /dashboard/candidate       Real-time own vote count (candidateId derived from session)

EC
  /admin/elections           List elections; create new
  /admin/elections/[id]      Setup: categories, candidates, times; PUBLISH button when closed
  /admin/elections/[id]/live Live dashboard: all counts real-time + audit log export button
  /admin/students            Manage student accounts; bulk CSV import

HOD
  /admin/live                Read-only live dashboard; auto-selects active election

SHARED (no auth required when published)
  /results/[electionId]      Full results when status === 'published'; accessible
                             without login (unauthenticated callers see full tallies);
                             authenticated students also use this route post-publish.
                             NOT listed under STUDENT ‚Äî one route, shared.
```

**Election status state machine (5-state, matching design doc):**
```
draft --[EC locks setup: markReady()]--> ready
ready --[cron: startTime <= now() OR EC manually opens]--> active
active --[cron: endTime <= now() OR EC manually closes]--> closed
closed --[EC clicks PUBLISH: publishElection()]--> published
```
Canonical DB status strings: `{ 'draft', 'ready', 'active', 'closed', 'published' }`.
`activateElection(electionId)` is called by the cron when `now() >= startTime && status === 'ready'`. Asserts no other election has `status === 'active'`; if one exists, throws. EC cannot manually bypass this constraint.
- `draft ‚Üí ready`: requires at least 1 category, 1+ candidates per category, valid startTime and endTime. Calling `markReady()` locks setup.
- `endTime` is immutable once status >= 'active'. EC can trigger early close (sets `earlyClosedAt`, transitions to `closed`).
- All `addCandidate` / `removeCandidate` / `addCategory` / `removeCategory` mutations must assert `status === 'draft'` before writing. Once `active`, setup is locked ‚Äî no candidate removal possible.
- Any user with `role === 'EC'` can manage any election (no per-election ownership).
- HOD `/admin/live`: auto-selects the single election where `status === 'active'`; if none, shows an empty state ("No active election"); if more than one, shows the most recently created (the system enforces only one can be active at a time via `activateElection`).

**Role-gated electionResults query (server-enforced):**
- `role === 'EC'` or `'HOD'`: full live counts regardless of status
- `role === 'student'`, not a candidate: counts only if `status === 'published'`; otherwise returns `{ status: 'voting_in_progress' }` (this is a response value, not a DB status string)
- `role === 'student'`, is a candidate (check `candidates` table by `userId` ‚Äî NOT `users.role`): same as above + their own `candidate_tally` count always; candidateId derived by querying `candidates` where `studentId === currentUser._id`; if no row, user is not a candidate
- Unauthenticated (no session, `ctx.auth.getUserIdentity()` returns null): counts only if `status === 'published'`, else `{ status: 'not_published' }`

### CEO plan expansions
- `v.optional(v.id("_storage"))` field `photoStorageId` in `candidates` table (design doc said V2-only; CEO plan accepts it now to avoid migration debt ‚Äî explicit override)
- `/dashboard/candidate` is in scope (in design doc screen map; candidate real-time own count via `myVoteCount()` ‚Äî no args, session-derived; server looks up `candidates` table by `userId`, returns own tally; if caller has no row in `candidates`, returns null)
- EC-only `auditLogExport` query: returns voted_log entries for a closed/published election. CSV columns: `["studentId", "categoryId", "timestamp"]` ‚Äî no candidate link exposed. Triggered by "Export Audit Log (CSV)" button visible only when election status is `closed` or `published`. Button lives on `/admin/elections/[id]/live`. No separate results CSV in V1.
- `/results/[electionId]` route: readable without auth when election status is `published`; unauthenticated callers (`ctx.auth.getUserIdentity()` returns null) get full tallies. Authenticated callers of any role also see full tallies. Note: `published` is the gate for this route ‚Äî not `closed`. Authenticated students also wait for `published` before seeing results on this route.

```
  - Add convex-test to devDependencies: `bun add -d @convex-dev/test` ‚Äî not

  - ConvexClientProvider must wrap layout.tsx ‚Äî verify the Next.js 16 App Router
    pattern (may differ from older Next.js docs).

HOUR 2-3 (core logic):
  - `tokenIdentifier` format with Convex Auth password provider: may include a
    provider prefix (e.g. "password:email@example.com"). The `by_token` index
    on users must use the exact format Convex Auth sets ‚Äî verify before castVote.
  - castVote is a single Convex mutation. The entire sequence (check voted_log,
    insert voted_log, patch candidate_tallies) runs in one mutation. Convex
    mutations are serialized ‚Äî no additional locking is needed. Do NOT split into
    multiple mutations (race condition risk).
  - castVote checks voter.isFirstLogin === false server-side (return
    { status: 'must_change_password' }). This is separate from the middleware
    redirect ‚Äî both guards are required.
  - addCandidate mutation: inserts { candidateId, categoryId, count: 0 } into
    candidate_tallies. castVote assumes this row exists and throws if missing.
    completeFirstLogin(newPassword) mutation: hashes new password, sets
    isFirstLogin = false, and forces session re-issue (do not just redirect).
  - First-login: middleware-level redirect prevents bypass; server-side castVote
    guard is the second enforcement layer.

HOUR 4-5 (integration):
  - myVotingProgress returns Id<"categories">[] ‚Äî client marks matched categories
    as done. Handle empty array (no votes yet) as valid initial state in UI.
  - Public results route: unauthenticated callers get null from
    ctx.auth.getUserIdentity(). Query handles null explicitly: return tallies if
    status === 'published', else { status: 'not_published' }.
  - Cron runs every 60s ‚Äî elections go active/closed up to 59 seconds late.
    castVote's server-side now() < endTime check is the hard guard. EC live
    dashboard should show "Scheduled to close at [time]" so ECs are not confused
    when status does not flip immediately at endTime.

HOUR 6+ (polish/tests):
  - by_voter_category index on voted_log: easy to forget. Without it, the
    duplicate-vote check is a full table scan. Verify it's in schema.ts.
  - CSV export: browser-side only (papaparse or manual string build) ‚Äî no
    server-side file generation needed. Columns: studentId, categoryId, timestamp.
  - Test the already_voted path ‚Äî it's the core integrity guarantee.
  - Test the unauthenticated public results path: null identity + status ===
    'published' ‚Üí full tallies. Null identity + status !== 'published' ‚Üí not_published.
```

## Reviewer Concerns (unresolved after 3 spec review iterations)

These items did not block the review score above 7.5/10 ‚Äî each is real but addressable during implementation:

1. **Session re-issue after first-login:** "Force session re-issue" is underspecified. During implementation, verify whether Convex Auth's password provider requires calling `invalidateUserSessions()` or whether signing in again with the new password is sufficient. Resolve in Hour 2-3 of implementation.

2. **Post-last-vote redirect:** What happens when a student submits their final category vote? Recommendation: redirect to `/vote` with a "All votes cast" banner; if election closes before they finish, redirect to `/results/[electionId]`.

3. **`createElection` mutation spec:** `createElection(title, description, startTime, endTime)` ‚Äî EC-only, creates with `status: 'draft'`. `startTime`/`endTime` required at creation; editable until `markReady()` is called.

4. **HOD route guard (middleware):** Middleware should redirect HOD to `/admin/live` for any route outside `/admin/live` and `/results/*`. HOD cannot access EC admin routes.

5. **Approach B (Clerk) stub:** If Convex Auth `createAccount` spike fails, the Clerk pivot replaces `users.passwordHash` with `users.clerkId`, and `ctx.auth.getUserIdentity()` returns Clerk JWT claims. First-login enforcement moves to a Convex `users.isFirstLogin` flag checked the same way. Session re-issue is Clerk's `signOut()` + `signIn()`.

6. **Late-vote UX (`castVote` returns `election_closed`):** When `now() >= endTime`, `castVote` should return `{ status: 'election_closed' }`. Client shows a toast "Voting has ended" and redirects to `/results/[electionId]` or a waiting screen if status is not yet `published`.

7. **`/dashboard/candidate` screen spec:** One number ‚Äî candidate's current tally, live. No voting UI. Nav link back to `/vote` if election is still active.

## Engineering Review Amendments
_Added by /plan-eng-review 2026-05-21 ‚Äî all decisions confirmed by user._

### Schema changes
- **`candidate_tallies` table ELIMINATED** ‚Äî `count: number` (default 0) added directly to `candidates` table. Removes the "remove deletes two rows" complexity and the "missing tally row" failure mode. Privacy partition is intact (candidates never has voter info).
- **`ec_action_log` table ADDED** ‚Äî `{ electionId, action, actorId (ref users), timestamp, metadata }`. Mutations that insert: `earlyClose`, `publishElection`, `removeCandidate`, `addCandidate`. Visible to EC on `/admin/elections/[id]/live`. Closes the "EC action audit" gap in the "indisputable" claim.

### Security changes
- **Default password: random initial token** (NOT studentId). `bulkImportStudents` generates a random 8-12 char alphanumeric token per student. After import, EC downloads a credentials CSV: `name, studentId, email, initialPassword`. Eliminates the account-takeover window during the pre-election period.

### Auth/route protection
- **`middleware.ts`**: `convexAuthNextjsMiddleware()` ‚Äî session existence only. No role check.
- **Route group layouts** (Server Components): `fetchQuery(api.users.getMe)` ‚Üí redirect if wrong role. `isFirstLogin` redirect in `/vote` layout.
- **`completeFirstLogin` mutation** updates `passwordHash + isFirstLogin=false`. Client immediately calls `signIn({ email, password: newPassword })` for fresh session token. Redirect to `/vote`.

### Code quality additions
- **`convex/lib/auth.ts`**: `getUser(ctx)` helper ‚Äî extracts `getUserIdentity ‚Üí users.by_token lookup ‚Üí return user`. Used in 6+ mutations. Bundled into this PR (was TODOS.md P1 pre-implementation).
- **`convex/lib/types.ts`**: `VoteStatus`, `ElectionResultStatus` discriminated unions. All mutation return validators reference these types.

### `removeCandidate` clarified
- Deletes `candidates` row only (count is now on the candidates row, so no second table). Status must be `draft`. EC action logged.
